# 变更提案: 手机号/银行卡号码类标签（存在/提取/无效/遮挡）99%+可审计改造

## 需求背景
当前项目的核心使用场景是对警情/接警自由文本做自动化标签分析。号码类信息（身份证号/手机号/银行卡号）具有以下共同特征：

- 数据来源复杂：同一条文本中可能混杂时间戳、警号、案号、金额、流水号等长数字串
- 号码经常被脱敏/遮挡：例如 `6212***********3290`、`1553***5598`，不能简单按“提取明文”验收
- 业务要求更偏“存在性与质量排查”：不仅要识别“正确号码”，也要识别“错误/无效/不完整但明显是号码”的候选

在身份证相关任务（例如 Task43：提取不满足18位的错误身份证）上，已通过“规则侧号码证据建模（SSOT）”修复了系统性漏检：避免“银行卡优先占用范围导致身份证候选被吞掉”。接下来需要把同样的工程化方法扩展到 **手机号** 与 **银行卡号**，并对前端“新建数据集标签”的配置能力做增强，以匹配新的标签定义与执行策略。

## 产品分析

### 目标用户与场景
- **用户群体:** 接警/警情数据分析人员、标注人员、质检人员、业务研发
- **使用场景:** 批量导入/同步警情文本 → 配置号码类标签 → 批量分析 → 复核导出 → 追踪误判原因并迭代规则/提示词
- **核心痛点:**
  - 号码混杂与遮挡导致“只提取有效明文”的方案不可用
  - LLM 直接抽取存在幻觉与不可审计问题，难以做到可回归的高准确
  - 前端创建标签时无法完整表达“号码类意图与输出策略”，导致后端无法稳定执行

### 价值主张与成功指标
- **价值主张:** 把“号码类标签”尽量变成可验证的确定性计算（规则SSOT），LLM 仅做语义解释/归属与辅助复核；全链路输出可审计证据。
- **成功指标（建议以回归集衡量）:**
  - 号码类标签（手机号/银行卡）在回归集上的 **precision/recall/F1 ≥ 0.99**（以“可验证口径”定义为准）
  - `needs_review` 占比可控（例如 ≤ 3%，由业务接受度决定）
  - 默认脱敏：日志/提示词/导出不出现完整号码（满足隐私合规与可控传播）

### 人文关怀（隐私与合规）
号码类信息属于敏感信息（PII）。本方案默认脱敏展示，并且对遮挡号码不尝试“还原明文”，只输出遮挡证据与存在性判断，避免二次扩散风险。

## 变更内容
1. **号码证据SSOT扩展（规则侧）**：在现有 `NumberEvidenceExtractor` 基础上，补齐手机号/银行卡的 valid/invalid/masked 证据与派生字段（count/exists），并形成统一可审计的证据输出。
2. **“号码类标签意图（number_intent）”显式化**：不再依赖 description 的自然语言猜测分支；通过配置声明 `entity`（phone/bank_card/id_card）、`task`（exists/extract/invalid/masked）与 `include`（valid/invalid/masked）等，使规则可完全接管可验证任务。
3. **前端“新建数据集标签”升级**：提供号码类向导/模板，生成 `preprocessor_config.number_intent`；并补齐创建/更新时关键字段透传（例如 includePreprocessorInPrompt），确保创建与编辑行为一致。
4. **一致性校验与复核策略**：对需要 LLM 的场景（语义归属/解释/歧义裁决）启用“证据ID约束 + 规则一致性校验 +（可选）二次强化复核”，把幻觉风险降到可控范围。

## 影响范围
- **模块:**
  - backend：号码证据建模、标签意图解析与规则执行、一致性校验、任务执行链路
  - frontend：数据集标签创建/编辑表单（号码类意图配置、字段透传）
  - helloagents 知识库：新增/更新方案文档与验收口径
- **API:** `POST /labels`、`PUT /labels/{id}` 请求体需要支持前端配置透传（重点：includePreprocessorInPrompt、preprocessorConfig）
- **数据:** 优先复用 `labels.preprocessor_config` 存储 `number_intent`（JSON扩展，避免强制数据库迁移）

## 核心场景

### 需求: 手机号存在性（错误/遮挡也算存在）
**模块:** backend + frontend

#### 场景: 文本包含有效手机号（11位，1[3-9]开头）
- 预期结果：`phone_exists=是`；可提取到 valid 列表

#### 场景: 文本包含无效手机号（11位，但号段不合法）
- 预期结果：`phone_exists=是`；可提取到 invalid 列表；`phone_invalid=是`

#### 场景: 文本包含遮挡手机号（包含*的11位片段）
- 预期结果：`phone_exists=是`；`phone_masked=是`；不尝试还原明文，仅输出遮挡证据（maskedValue/maskPattern）

### 需求: 银行卡号存在性/提取/无效判断（遮挡也算存在）
**模块:** backend + frontend

#### 场景: 16-19位数字且通过 Luhn
- 预期结果：`bank_exists=是`；提取 valid；并保留校验信息（Luhn通过）

#### 场景: 16-19位数字但 Luhn 不通过（且具银行卡弱特征/语境）
- 预期结果：`bank_exists=是`；提取 invalid；并保留“为何判为银行卡无效”的证据（前缀弱特征/关键词窗）

#### 场景: 银行卡号被遮挡（16-19位包含*）
- 预期结果：`bank_exists=是`；`bank_masked=是`；不输出不可验证明文，仅输出遮挡证据

### 需求: 标签定义与创建能力升级（号码类向导）
**模块:** frontend + backend

#### 场景: 用户新建“手机号存在性”标签（规则模式）
- 预期结果：前端生成 `number_intent` 配置并落库；后端纯规则输出“是/否”，无需 LLM

#### 场景: 用户新建“银行卡无效号码提取”标签（规则模式）
- 预期结果：后端输出可审计结果（建议返回“号码列表 + 证据摘要/原因”），并默认脱敏展示

## 风险评估
- **口径风险（关键）:** 若把“无效/错误”理解为“真实不存在/不可用/归属不符”，仅靠文本与LLM不可验证；本方案默认“无效=格式/校验失败/不可验证（遮挡）”。
- **误判风险:** 长数字串（案号/时间戳/流水）可能被误判为银行卡或手机号；需“结构特征 + 上下文关键词窗 + 冲突仲裁”组合降误判。
- **隐私风险（EHRB）:** 号码类为敏感信息，必须默认脱敏；需要明文的场景必须受权限与审计控制。
- **准确率承诺风险:** “99%+”必须绑定可验证口径与回归集覆盖范围；对遮挡/缺失/歧义样本应允许 `needs_review`，避免为了“自动出结论”而牺牲正确性。
