# 涉警当事人信息完整性检查_v2 - 修正版提示词

## 核心原则
严格遵循原始标签定义，不添加额外要求（如公司营业执照检查），保持逻辑简洁。

## 优化后的提示词

```markdown
## 任务：涉警当事人信息完整性检查

### 标签定义
检查警情中涉警当事人身份信息是否完整（身份证号/护照号）。

### 涉警当事人范围
- **纳入**：纠纷双方、侵权行为人、受害人等与警情事实直接相关人员
- **排除**：处警民警/辅警、围观群众、与警情无关路人等

### 直接合格情形
若文本包含以下关键词，可直接判定为"是"，无需继续检查当事人信息：
- "请求撤警"、"要求撤警"、"自愿撤警"
- "误报警"、"重复警情"、"无效警情"
- "无事实"、"未发现报警情况"、"无需处理"

### 合格标准
**所有涉警当事人均应提供有效身份号码**：
- 18位身份证号（格式正确）
- 或护照号

**同名处理**：若同一姓名出现多次，任一处出现有效身份证号/护照号视为该人员信息完整。

### 不合格标准
存在任一涉警当事人身份证号缺失/不完整/格式错误，且无法以护照号等其他有效信息补齐。

### 判定步骤

#### 第一步：检查是否属于直接合格情形
- 搜索撤警、无效警情等关键词
- 如果是，直接返回"是"，结束判定

#### 第二步：识别涉警当事人
从文本中识别**所有**涉警当事人（纠纷双方、侵权人、受害人等）：
- 报警人/报案人
- 对方当事人（纠纷对方、侵权人）
- 受害人/被侵权人
- ⚠️ 排除：处警民警/辅警、围观群众、无关路人

#### 第三步：提取有效身份信息
对每个当事人，检查是否提供了：
- 18位身份证号（格式正确，无前缀/后缀字符）
- 或护照号

**格式错误判定**：
- 身份证号前有`？`、`#`等字符 → 格式错误
- 护照号格式异常 → 格式错误

#### 第四步：同名合并
如果同一姓名出现多次，只需其中一处有有效证件即可。

#### 第五步：最终判定
- **判定为"是"**：所有涉警当事人都有有效身份证号或护照号
- **判定为"否"**：存在任一当事人身份证号缺失/不完整/格式错误

### 输出格式
```json
{
  "conclusion": "是" 或 "否",
  "confidence": 1-10,
  "reasoning": "判定理由",
  "is_withdrawal": true/false,
  "parties": [
    {"name": "当事人姓名", "has_id": true/false}
  ],
  "missing_party": "缺失信息的当事人姓名"
}
```

### 特别注意
1. 不要找到一个身份证就判定"是"，必须检查**所有**当事人
2. 同名人员只需一处有证件即可
3. 不要将处警人员、围观群众计入涉警当事人
```

## v1 vs v2 对比

| 方面 | v1问题 | v2修正 |
|-----|--------|--------|
| 步骤流程 | 无明确步骤 | 5步严格执行流程 |
| 同名处理 | 未提及 | ✅ 明确"同名合并"规则 |
| 排除人员 | 未强调 | ✅ 明确排除民警/辅警/围观群众 |
| 公司主体 | 无要求 | ⚠️ **不添加额外要求**，保持原始定义 |
| 输出结构 | 无固定结构 | ✅ JSON结构化 + parties列表 |
| 常见错误 | 无警示 | ✅ 添加特别注意 |

## 测试用例

| 场景 | 输入 | 期望 |
|-----|------|------|
| 正常-单人 | 报警人张三（3401...） | 是 |
| 正常-双人 | 报警人张三（3401...），对方李四（3402...） | 是 |
| 错误-单人证件 | 报警人张三（3401...），对方李四未提供证件 | 否 |
| 错误-前缀字符 | 报警人张三（？3401...） | 否 |
| 撤警情形 | 报警人请求撤警，其他信息不详 | 是 |
| 同名处理 | 张三（3401...），另一次提到张三无证件 | 是（同名合并） |
| 排除人员 | 报警人张三，处警民警李四，围观群众王五 | 仅张三需检查 |
```
