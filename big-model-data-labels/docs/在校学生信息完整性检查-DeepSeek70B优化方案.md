# 在校学生信息完整性检查 - DeepSeek 70B 优化方案

## 一、问题诊断

### 1.1 当前问题分析

根据44条错误数据的分析，主要问题包括：

| 错误类型 | 占比 | 根本原因 |
|---------|------|---------|
| 辍学学生判定错误 | 38.6% | 标签定义中"辍学/休学仍需标明学籍"存在歧义 |
| 学校名称不完整 | 54.5% | 缺少学校全称验证规则 |
| 未识别学生身份 | 6.8% | 未通过年龄推断学生身份 |

### 1.2 DeepSeek 70B 限制

1. **长文本理解能力有限**：警情描述通常很长（列14可达1000+字符）
2. **信息遗漏**：在长文本中容易忽略关键细节
3. **推理能力不足**：复杂逻辑判断容易出错

---

## 二、优化方案

### 2.1 标签定义优化

#### 当前定义（有歧义）
```
【在校学生定义】国家教育行政部门批准设立的教育机构注册的全日制在读学生。
辍学/休学仍需标明学籍。
```

#### 优化后定义（明确）
```
【在校学生定义】国家教育行政部门批准设立的教育机构注册的全日制在读学生。

【明确不属于在校学生】
1. 辍学学生：已明确标注"辍学"、"退学"、"已毕业"的学生
2. 成人教育：成人教育、函授、自考等非全日制学生
3. 培训机构：社会培训机构、补习班、辅导班学员

【特殊情况】
- 休学学生：保留学籍的休学学生仍属于在校学生，但需标注"休学"状态
- "标明学籍"的含义：需要记录学生曾在哪所学校就读，不等于仍在读

【判断原则】
- 已辍学 + 有学籍信息 → 不涉及在校生 → 判"是"
- 在校生 + 信息完整 → 判"是"
- 在校生 + 信息不完整 → 判"否"
```

### 2.2 提示词优化（适配 DeepSeek 70B）

#### 优化策略
1. **分段处理**：将长文本拆分为"人员信息段"和"警情描述段"
2. **结构化输出**：要求模型按步骤输出，便于验证
3. **关键检查点**：明确列出必须检查的5个关键点

#### 优化后的主提示词

```mustache
你是在校学生信息审核专家。请检查警情中是否涉及在校学生，以及信息是否完整。

========== 第一步：识别所有在校学生 ==========
请逐个人员判断是否属于"在校学生"：

【在校学生标准】
✅ 属于：幼儿园/小学/初中/高中/中职/高校在读学生
❌ 不属于：辍学、毕业、成人教育、培训机构学员

【关键检查点】
1. 是否明确标注"学生"？
2. 是否标注"辍学"、"毕业"？→ 不属于在校生
3. 是否标注"培训机构"、"函授"、"成人教育"？→ 不属于在校生
4. 年龄6-18岁且未标注"非学生"？→ 推断为在校生
5. 学校名称是什么？是否包含省/市前缀？

【输出格式 - 人员列表】
对每个人员，输出：
- 姓名：
- 年龄：
- 是否在校生：（是/否/推断为在校生）
- 学校名称：
- 判断依据：

========== 第二步：检查信息完整性 ==========
如果第一步识别到在校生，请检查6项信息是否完整：

【6项必需信息】
1. 姓名
2. 身份证号（必须18位）
3. 学校全称（必须包含省/市，如"安徽省XX中学"）
4. 在读年级（如"初一"、"2023届"等）
5. 院系/专业（小学/初中/高中可不要求，中职/高校必须）
6. 联系方式（手机号）

【学校全称验证】
- ❌ 简称不合格：如"北师大"、"安农"等简称
- ❌ 缺少省/市不合格：如"桐城市第八中学"缺少"安徽省"
- ✅ 正确示例：安徽省桐城市第八中学、安徽农业大学

【输出格式 - 信息完整性】
对每个在校生，输出：
- 姓名：
- 缺少信息项：（如无则填"无"）

========== 第三步：最终判断 ==========
根据前两步的分析，给出最终判断：

【判断规则】
- 不涉及在校生 → "是"
- 涉及在校生且6项信息全部完整 → "是"
- 涉及在校生但有任意信息缺失 → "否"

【输出格式 - 最终结果】
{
  "result": "是/否",
  "reasoning": "详细的推理过程",
  "persons": [
    {
      "name": "姓名",
      "is_student": true/false,
      "missing_items": ["缺少的信息项"]
    }
  ],
  "confidence": 80
}
```

【核心错误2：学校名称不完整】
- 检查学校名称是否包含省/市前缀
- 检查是否存在简称（如"北师大"、"安农"等）

【核心错误3：6项信息缺失】
- 逐项检查：姓名、身份证号、学校全称、年级、院系/专业、联系方式
- 小学/初中/高中可不要求院系/专业
- 中职/高校必须提供院系/专业

【核心错误4：未识别学生身份】
- 检查6-18岁人员是否被识别为在校生
- 如果年龄合理但未识别为在校生，可能误判

【核心错误5：培训机构误判】
- 检查是否存在"培训机构"、"函授"、"成人教育"等关键词
- 这些不属于在校生

### 2.4 规则后置验证器

已创建 `StudentInfoValidator.java`，专门针对在校学生信息的后置验证。

#### 核心功能
1. **辍学检测**：检测"辍学"、"退学"、"毕业"关键词
2. **学校全称验证**：检测简称和缺少省/市前缀的情况
3. **培训机构检测**：检测培训机构关键词
4. **6项信息完整性检查**：逐项验证必需信息
5. **年龄推断**：对6-18岁人员推断学生身份

#### 使用方式

```java
@Autowired
private StudentInfoValidator studentValidator;

public ValidationResult validateStudentInfo(String result, String text) {
    return studentValidator.validate(result, text);
}
```

## 三、实施步骤

### 3.1 立即实施（高优先级）

1. **修改标签定义**
   - 更新 `BuiltinGlobalLabelsInitializer.java` 中的 `buildStudentLabelDescription()`
   - 明确"辍学"学生不属于在校生

2. **集成 StudentInfoValidator**
   - 在 `EnhancementService` 中集成 `StudentInfoValidator`
   - 在后置验证阶段调用

### 3.2 短期实施（中优先级）

3. **更新提示词**
   - 更新 `buildStudentEnhancementTemplate()`
   - 增加关键检查点和结构化输出要求

4. **优化触发条件**
   - 在 `EnhancementConfig` 中增加强制触发条件
   - 检测到"辍学"、"简称"等关键词时强制触发二次验证

### 3.3 长期优化（低优先级）

5. **建立学校名称库**
   - 收集安徽省所有学校的官方名称
   - 用于验证学校名称是否完整

6. **分段处理优化**
   - 实现长文本分段处理逻辑
   - 优先提取人员信息，再提取详细信息
