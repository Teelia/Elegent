# 涉警当事人信息完整性检查标签优化方案

> **版本**: v1.0
> **日期**: 2026-01-28
> **基于**: Task 74 错误分析结果（48条错误结论）

---

## 一、问题诊断总结

### 1.1 错误类型分布

| 错误类型 | 数量 | 占比 | 典型案例 |
|---------|------|------|---------|
| 身份证号格式错误误判 | 2 | 4% | 行号0（19位）、行号144（17位） |
| 遗漏涉警当事人 | 30+ | 62% | 行号9（小孩）、行号52（中介）、行号95（收费方） |
| 信息缺失关键词未识别 | 10+ | 21% | 行号707（拒绝透露）、"研判无果"等 |
| "直接合格"误用 | 6 | 12% | 将撤警关键词误判为无需检查完整信息 |

### 1.2 根本原因分析

#### A. 配置层面
- **预处理配置宽松**：`id15IsValid: true` 导致15位身份证被接受
- **长度验证缺失**：没有明确拒绝17位、19位等非标准长度
- **二次强化阈值过高**：`triggerConfidence: 75` 使多数错误案例跳过二次验证

#### B. 提示词层面
- **验证指令不明确**：未明确要求"严格18位验证"
- **当事人识别范围窄**：仅关注明确标注的"报警人/当事人"
- **关键词敏感性不足**：未强调"拒绝"、"未提供"、"研判无果"的否定意义

#### C. 逻辑层面
- **涉警当事人定义模糊**：缺少对"纠纷相关方"的语义理解
- **豁免条件误用**：直接合格关键词覆盖了信息完整性检查

## 二、优化方案总览

### 2.1 优化目标
- **准确率提升**：从当前 ~93% 提升至 ≥98%
- **误判率降低**：将格式错误误判率降至 <1%
- **召回率保持**：确保合格案例不被误判为不合格

### 2.2 优化策略
```
┌─────────────────────────────────────────────────────────────┐
│                    优化策略金字塔                            │
├─────────────────────────────────────────────────────────────┤
│  L1: 基础规则增强          (身份证号严格验证)                │
│  L2: 预处理优化            (当事人识别 + 缺失检测)           │
│  L3: 提示词强化            (明确指令 + 上下文理解)          │
│  L4: 二次强化改进          (降低阈值 + 精准复核)             │
│  L5: 质量保障机制          (规则引擎后置验证)                │
└─────────────────────────────────────────────────────────────┘
```

**新增号码长度验证器**

```java
/**
 * 身份证号长度严格验证器
 * 仅接受18位（前17位数字 + 最后1位数字或X）
 */
public class IdCardLengthValidator {
    private static final Pattern VALID_ID_PATTERN = Pattern.compile("^\\d{17}[0-9Xx]$");

    public static ValidationResult validate(String idNumber) {
        if (idNumber == null || idNumber.length() != 18) {
            return ValidationResult.invalid(
                String.format("身份证号长度错误：%d位（必须18位）", idNumber == null ? 0 : idNumber.length())
            );
        }
        if (!VALID_ID_PATTERN.matcher(idNumber).matches()) {
            return ValidationResult.invalid("身份证号格式错误：前17位必须为数字，第18位为数字或X");
        }
        return ValidationResult.valid();
    }
}
```

```java
public boolean shouldTrigger(int initialConfidence, String validationResult) {
    // 基础触发条件
    if (initialConfidence < triggerConfidence) {
        return true;
    }

    // ✅ 新增：强制二次验证条件
    if (validationResult != null) {

        if (validationResult.contains("格式错误") ||
            validationResult.contains("长度错误") ||
            validationResult.contains("17位") ||
            validationResult.contains("19位")) {
            return true;

            validationResult.contains("未提供") ||
            validationResult.contains("未登记") ||
            validationResult.contains("研判无果")) {
            return true;

        // 涉警当事人数量不匹配
        if (validationResult.contains("当事人") &&
            validationResult.contains("缺失")) {
            return true;
        }
    }

    return false;
}
```

### 3.5 L5: 质量保障机制

#### 新增：规则引擎后置验证

```java
/**
 * 后置规则验证器
 * 对LLM判定结果进行规则层面的最终验证
 */
@Service
@Slf4j
public class PostProcessValidator {

    /**
     * 验证判断结果是否符合规则
     *
     * @param result LLM判断结果
     * @param extractedNumbers 提取的号码
     * @param missingInfoFlags 信息缺失标记
     * @return 验证结果
     */
    public ValidationResult validate(String result,
                                     List<ExtractedNumber> extractedNumbers,
                                     MissingInfoFlags missingInfoFlags) {

        // 规则1：如果LLM判定为"是"，必须严格验证
        if ("是".equals(result)) {
            return validatePositiveResult(extractedNumbers, missingInfoFlags);
        }

        // 规则2：如果判定为"否"，检查是否有误判风险
        return validateNegativeResult(extractedNumbers, missingInfoFlags);
    }

    private ValidationResult validatePositiveResult(List<ExtractedNumber> numbers,
                                                    MissingInfoFlags flags) {
        List<String> errors = new ArrayList<>();

        // 1. 检查身份证号格式
        for (ExtractedNumber num : numbers) {
            if ("id_card".equals(num.getField())) {
                ValidationResult vr = IdCardLengthValidator.validate(num.getValue());
                if (!vr.isValid()) {
                    errors.add(String.format("身份证号 %s 格式错误：%s",
                        num.getValue(), vr.getReason()));
                }
            }
        }

        // 2. 检查信息缺失标记
        if (flags.hasMissingInfo()) {
            errors.add("检测到信息缺失标记：" + flags.getMissingKeywords());
        }

        // 3. 检查当事人完整性
        if (flags.hasIncompletePerson()) {
            errors.add("存在涉警当事人信息不完整：" + flags.getIncompleteReasons());
        }

        if (!errors.isEmpty()) {
            return ValidationResult.invalid(String.join("; ", errors));
        }

        return ValidationResult.valid();
    }

    private ValidationResult validateNegativeResult(List<ExtractedNumber> numbers,
                                                    MissingInfoFlags flags) {
        // 检查是否可能误判
        boolean allValid = numbers.stream()
            .filter(n -> "id_card".equals(n.getField()))
            .allMatch(n -> IdCardLengthValidator.validate(n.getValue()).isValid());

        if (allValid && !flags.hasMissingInfo() && !flags.hasIncompletePerson()) {
            return ValidationResult.warning(
                "所有身份证号格式有效且无信息缺失标记，可能存在误判风险"
            );
        }

        return ValidationResult.valid();
    }
}
```

## 九、总结

本优化方案通过 **5层递进式优化**，系统性地解决了涉警当事人信息完整性检查标签的4大类问题：

1. **L1 规则层**：严格18位验证，拒绝非标准长度
2. **L2 预处理层**：增强当事人识别和信息缺失检测
3. **L3 提示词层**：明确指令，强调关键检查点
4. **L4 强化层**：降低阈值，增加强制触发条件
5. **L5 保障层**：后置规则验证，防止LLM误判

预期效果：**准确率从93%提升至98%**，误判数从48条降至≤10条。
