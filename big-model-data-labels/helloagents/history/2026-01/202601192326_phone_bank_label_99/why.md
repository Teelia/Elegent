# 变更提案: 手机号/银行卡号码类标签（存在/提取/错误无效）99%+可审计改造

## 需求背景
当前项目已在身份证相关任务（如“不满足18位的错误身份证号”）上验证了“规则证据建模（SSOT）→（可选）LLM受限分析”的可行性，并通过回归集消除了系统性漏检。

接下来需要把同样的工程化思路扩展到 **手机号** 与 **银行卡号**：
- **存在性**：只要文本里出现（即使错误/无效/被遮挡），也应判定“存在”。
- **提取**：输出可审计的号码列表（建议区分有效/无效/遮挡）。
- **错误/无效判断**：对可验证规则（号段/长度/Luhn 等）给出确定性结论；对遮挡样本给出可复核结论（如 masked / needs_review）。

同时，前端“新建数据集标签”的属性与后端创建标签的DTO存在不一致，导致部分配置在创建时无法落库（只能更新时生效），不利于按“SSOT证据→一致性校验”的方案推进。

---

## 产品分析

### 目标用户与场景
- **用户群体**：接警/警情数据分析与标注人员、质检人员、业务研发。
- **使用场景**：批量导入/同步警情文本，针对身份证/手机号/银行卡等敏感号码做存在性判断、抽取、错误排查、复核导出。
- **核心痛点**：
  - 号码字段经常被脱敏（`****`）或不完整，纯“提取明文”无法满足“存在性”要求。
  - 长数字串混杂（时间戳/警号/案号/流水），误判风险高；一旦误判会显著降低可信度。
  - LLM 直接抽取存在“幻觉”，缺乏可审计证据链，难以做质量回归与持续优化。

### 价值主张与成功指标
- **价值主张**：把号码类任务尽量变成“可验证的确定性计算”，LLM 仅用于语义解释/归因/归属，输出全链路可审计。
- **成功指标（建议）**：
  - 号码类标签（手机/银行卡）在回归集上的 **precision/recall/F1 ≥ 0.99**（以“可验证口径”定义为准）。
  - `needs_review` 占比可控（例如 ≤ 3%，由业务接受度决定）。
  - 全链路默认脱敏输出，避免 PII 泄露。

### 人文关怀（隐私与合规）
默认不在日志/提示词/导出中暴露明文号码；对必须展示明文的场景，需要在权限与审计下开放（避免二次扩散）。

---

## 变更内容
1. **号码证据SSOT扩展（规则侧）**：在现有 `NumberEvidenceExtractor` 基础上扩展手机号/银行卡的 valid/invalid/masked 证据类型与派生字段，形成可审计证据JSON。
2. **标签定义升级（“意图”显式化）**：不再依赖自然语言 description 猜测任务类型；通过标签配置（JSON）明确：
   - 目标实体：phone / bank_card
   - 目标任务：exists / extract / invalid / masked
   - 输出策略：只输出 valid / 输出 invalid / 仅存在性 / 是否允许 masked 证据等
3. **前端“新建数据集标签”改造**：提供“号码类标签向导/模板”，用结构化表单生成上述配置；并补齐创建/编辑时缺失的关键属性（如 includePreprocessorInPrompt）。
4. **质量回归与评估**：建立手机号/银行卡的回归集（xlsx）与指标脚本；对无法达到99%的口径，输出评估报告与可接受的降级策略（needs_review/仅存在性）。

## 影响范围
- **模块**：
  - backend：号码证据提取、标签配置解析与执行、LLM提示词证据约束、一致性校验、创建标签DTO/落库逻辑
  - frontend：数据集标签创建/编辑表单（新增“号码类意图配置”）
  - docs/helloagents：补充方案与口径、风险评估与验收说明
- **数据**：优先复用现有 `labels.preprocessor_config` / `labels.preprocessing_mode` 等字段（JSON扩展），避免强制数据库迁移；如后续需要可再引入专用字段。
- **API**：创建标签接口需支持完整字段（当前创建DTO缺失部分字段）。

## 核心场景

### 需求: 手机号存在性（错误也算存在）
**模块:** backend + frontend

#### 场景: 文本包含有效11位手机号
- 预期结果：exists=是；extract.valid 含该号码

#### 场景: 文本包含疑似手机号但不满足规则（长度/号段）
- 预期结果：exists=是；extract.invalid 含该号码；invalid=是

#### 场景: 文本中手机号被遮挡（如 `13****1234` / `1**********`）
- 预期结果：exists=是；masked=是；如无法恢复明文则不输出明文号码

### 需求: 银行卡号存在性/提取/无效判断
**模块:** backend + frontend

#### 场景: 16-19位数字且通过 Luhn
- 预期结果：exists=是；extract.valid 含该号码

#### 场景: 16-19位数字但 Luhn 不通过（且具银行卡语境或特征）
- 预期结果：exists=是；extract.invalid 含该号码；invalid=是

#### 场景: 银行卡号被遮挡（如 `62****5678`）
- 预期结果：exists=是；masked=是；不输出不可验证的明文

## 风险评估
- **口径风险（关键）**：若把“无效”定义为“真实不存在/无法使用”，仅靠文本与LLM不可验证；本方案默认“无效=格式/校验失败/不可验证（遮挡）”。
- **误判风险**：长数字串（案号/警号/订单号）可能被误认为手机号/银行卡；需要“结构特征 + 上下文特征（关键词窗）”共同裁决。
- **PII 风险（EHRB）**：号码类为敏感信息，必须默认脱敏、最小化存储与可审计访问。
