# 新建数据集标签（前端）简化改造建议

**适用页面**：`frontend/src/components/DatasetLabelManager.vue`（新建/编辑标签弹窗）  
**编写日期**：2026-01-20  

---

## 1. 现状问题（为什么“感觉很复杂”）
- **单弹窗承载过多能力**：分类/自由提取/结构化提取/规则预处理/混合模式/强化分析/JSON高级编辑/号码类向导全部堆在一个表单里，用户需要先理解架构才能正确配置。
- **关键路径不突出**：对“号码类标签”来说，最常见需求其实是：实体（身份证/手机号/银行卡）+ 任务（存在/提取/无效/遮挡…），但目前被“预处理模式/提取器/选项/JSON”淹没。
- **易误用组合**：例如启用了 `number_intent` 但仍被要求选择提取器；或用户选择 `rule_then_llm` 期望规则闭环，却不知道哪些字段最终生效。

---

## 2. 简化目标（不改后端数据模型也能达成）
1) 让用户在 30 秒内完成一个“号码类标签”的创建（不需要懂 JSON、不需要理解提取器体系）。  
2) 把“高频、低风险”的配置留在主路径；把“低频、高风险”的配置（JSON、动态选项、强化策略）收纳到高级区。  
3) 保持兼容：仍然保存同一套 `Label` 字段（type / preprocessingMode / preprocessorConfig / enhancementConfig）。  

---

## 3. 推荐交互方案（两层：简易 / 高级）

### 3.1 简易模式（默认）
**结构**：4 步（可用 Tabs 或 Steps）
1. 基础信息：名称 / 描述 / 关注列  
2. 标签类型（少选项）：  
   - 号码类（推荐）→ 进入 3)  
   - 非号码类（通用）→ 引导切到高级模式  
3. 号码类模板（强推荐）：选择下列模板之一（自动填充所有底层字段）  
   - 身份证：存在 / 提取 / 错误 / 遮挡 / 错误长度且遮挡  
   - 手机号：存在 / 提取 / 错误 / 遮挡  
   - 银行卡：存在 / 提取 / 错误 / 遮挡  
4. 一句话策略：  
   - 是否默认脱敏输出（默认是）  
   -（银行卡）无效号是否要求关键词窗命中（默认是）  

**简易模式下隐藏**：
- `preprocessorConfig` JSON 编辑
- 提取器列表与动态 extractorOptions（除非用户显式切到高级）
- 强化分析（默认关闭；如需，放到高级）

### 3.2 高级模式（完整）
保留现有能力，但做两点收纳：
- 用 `ElCollapse` 把“预处理器JSON/动态选项/强化分析”收起来，默认折叠。
- 当启用 `number_intent` 时，禁用/置灰“启用提取器”并给出提示：**向导已覆盖，无需选择提取器**。

---

## 4. 模板到落库字段的映射（SSOT）
以“号码类模板”为例（不新增后端字段）：
- `type`：推荐 `classification`（存在性）或 `extraction`（提取文本结果）  
- `preprocessingMode`：推荐 `rule_only`（规则闭环，稳定可回归）  
- `preprocessorConfig`：写入 `number_intent`（实体+任务+策略），无需用户手写 JSON  
- `includePreprocessorInPrompt`：仅在 `rule_then_llm` 时有意义（简易模式默认不暴露）  

---

## 5. 风险与控制
- **风险：用户看不到所有选项** → 提供“切换到高级模式”入口，并在高级模式保留全部字段。
- **风险：模板选择与历史标签不一致** → 模板只负责“默认值”，用户依然可在高级模式手动微调。
- **风险：不同模式下字段生效规则不清晰** → 在 UI 中明确标注“最终生效字段预览”（展示最终生成的 preprocessorConfig JSON，但只读）。

---

## 6. 建议实施顺序（低风险迭代）
1) 先做 UI 收纳：Collapse + 启用 number_intent 时置灰提取器列表  
2) 再做“号码类模板”下拉（自动填充）  
3) 最后做 Tabs/Steps 的结构化重排与组件拆分（可维护性提升）
