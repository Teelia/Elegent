# How

## 方案概述

采用“逐行审计 + 关键链路修复 + 回归测试”的方式：

1. **逐行审计报告生成**
   - 读取两份 xlsx 的 `分析结果` sheet。
   - 对每行提取：`行号/事件单编号/反馈内容/标签输出(含reasoning)`。
   - 解析标签输出文本中的：
     - 结论（是/否）
     - 后置验证摘要（通过/失败、错误原因）
     - 关键触发项（如：身份证前缀、信息缺失关键词、涉警当事人缺失等）
   - 生成一个新的审计文件（xlsx），并为每行标注“问题类型”和“修复后预期变化”。

2. **修复后置验证链路（labelName 透传）**
   - 在触发后置验证的位置，将 `label.getName()` 透传到 `PostProcessValidator.validate(..., labelName)`。
   - 保持对旧方法的兼容（如有外部调用），但在当前链路统一使用新签名。

3. **修复身份证空白前缀误判**
   - 调整身份证提取/前缀检测正则与逻辑：
     - 空白仅作为分隔符，不计入“前缀字符错误”。
     - 仅对明确的脏前缀（如 `?` `？` `#` `*` `＊`）进行“前缀错误”判定。

4. **降低当事人抽取噪声**
   - 对当事人名称做更严格的“人名样式”过滤（例如：2-4 个中文字符或显式“某某”），避免把动词短语/提示语当成当事人姓名。
   - 对“泛指当事人/对方不在现场”等场景，维持“信息不可验证”的处理，但避免污染缺失名单。

5. **回归测试**
   - 单元测试覆盖：
     - `IdCardLengthValidator`：空白前缀不应触发前缀错误；特殊字符前缀应触发。
     - 后置验证链路：在校学生标签应走 `StudentInfoValidator`。
     - `PartyExtractor`：过滤非人名片段，缺失名单不应出现“表示知晓”等噪声 token。

6. **知识库同步**
   - 更新 `helloagents/wiki/modules/backend.md` 与 `helloagents/CHANGELOG.md`，记录修复点、影响范围与回归方式。

## 风险与规避

- 风险：当事人抽取过滤过严导致漏检。
  - 规避：先“仅过滤明显噪声”，并通过审计报告对比修复前后差异，逐步调整。
- 风险：身份证正则调整影响其他标签。
  - 规避：以单元测试 + 小样本验证为门槛；仅修正“空白前缀”这一确定误判来源。
