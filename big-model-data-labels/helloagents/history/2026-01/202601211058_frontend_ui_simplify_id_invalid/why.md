# 变更提案: 前端交互重排与内置“错误身份证(长度错)”模板

## 需求背景
当前前端在“标签配置/提取器/预处理/JSON/向导/强化”等能力上堆叠在同一条路径中，导致学习成本高、误配置风险大。
本次改造希望在不牺牲现有高级能力的前提下，把主流程压缩为“少选项、可回归、可审计”的固定模板路径，并对全局布局、页面信息层级、交互反馈进行整体重排。

## 产品分析

### 目标用户与场景
- **用户群体:** 警情数据分析与标注人员、复核人员；以及少量管理员（维护提取器/模板/提示词）。
- **使用场景:** 快速创建标签→创建任务→查看进度→复核结果→导出；高频场景聚焦“号码类(身份证)”的存在/提取与错误识别。
- **核心痛点:**
  - “新建/编辑标签”一次性暴露过多字段（提取器/选项/JSON/向导/强化），用户需要先理解系统结构才能完成配置。
  - “提取器配置”过于靠前，普通用户被迫面对低频、专业的概念。
  - “结果复核”缺少证据闭环展示，复核人员无法快速判断系统结论是否可靠。

### 价值主张与成功指标
- **价值主张:**
  - 让普通用户通过内置模板在 30 秒内完成一个可用标签；同时保留高级模式满足复杂需求。
  - 让“号码判断”尽可能变成规则证据驱动，结果页可审计、可复核。
- **成功指标:**
  - 新建标签平均步骤减少（主路径 ≤ 4 步）；模板创建成功率提升。
  - 复核效率提升：结果页可直接查看“证据候选列表/规则摘要”，减少反复回看原文的次数。
  - 误配置下降：当启用模板/意图时，自动禁用或折叠冲突项（例如重复选择提取器）。

### 人文关怀
默认明文输出与证据展示会触及敏感信息（身份证号等）。在满足业务要求（默认明文）的同时，仍需提供“遮罩开关/权限提示/导出策略提示”，降低误泄露风险。

## 变更内容
1. 前端整体布局与信息架构重排：更清晰的导航、页面统一的“主动作区/内容区/侧信息区”。
2. 标签配置重构为“模板优先”的两级模式：简易模板（默认）+ 高级配置（折叠收纳）。
3. 内置固定模板：错误身份证号（长度错误）
   - 是否存在（classification）
   - 提取（extraction）
4. 结果复核新增“证据查看”：展示规则侧 SSOT（候选、类型、校验摘要、命中原因）。
5. 提取器配置页保留为全局可见，但默认折叠/收纳高级区，避免干扰主流程。

## 影响范围
- **模块:** frontend, backend
- **文件:**
  - 前端：App 布局、标签管理弹窗、提取器页、结果复核页等
  - 后端：number_intent 与证据输出口径（按最新业务口径调整）
- **API:** 可能新增/扩展标签结果接口以返回证据（或复用 extractedData 字段）
- **数据:** 不新增核心表；可能在 label_results.extracted_data 中写入证据结构（JSON）

## 核心场景

### 需求: 新建/编辑标签“模板优先”
**模块:** frontend
将“新建/编辑标签”交互重排为“模板优先 + 高级折叠”，默认隐藏低频配置。

#### 场景: 普通用户快速创建错误身份证(长度错)标签
在不理解提取器/JSON 的情况下，用户仅选择“模板：错误身份证(长度错)-是否存在/提取”，即可创建可用标签。
- 预期结果: 30 秒内完成创建；保存后生成稳定的 preprocessor_config（number_intent）。

### 需求: 结果复核展示规则证据（SSOT）
**模块:** frontend
在结果列表/详情中展示规则证据：候选列表（明文/脱敏切换）、类型、命中解释、needs_review。

#### 场景: 复核人员依据证据快速确认“错误身份证(长度错)”命中
复核人员不需要阅读长原文，只看证据抽屉即可判断该命中是否可靠。
- 预期结果: 每条结果可一键展开证据；支持复制/定位原文。

### 需求: 提取器配置全局可见但默认收纳
**模块:** frontend
提取器配置页保持入口，但在标签创建主路径中弱化提取器概念；配置项默认折叠在高级区。

#### 场景: 管理员维护提取器不影响普通用户
普通用户完成标签创建不需要进入提取器页；管理员仍可在该页维护自定义提取器。
- 预期结果: 主路径不出现“必须先配置提取器”的阻塞提示。

### 需求: 错误身份证号口径统一（长度错误）
**模块:** backend
按口径：仅长度错误算“错误身份证”；15位不算错；18位校验位不通过不算错；遮挡不参与。

#### 场景: 规则闭环输出明文并提供证据
规则输出默认明文；同时在 extracted_data 中输出证据结构供前端展示。
- 预期结果: exists 与 extract 都仅基于 ID_INVALID_LENGTH；不计 masked、不计 checksum。
