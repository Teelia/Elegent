## 二、问题分析

### 2.1 原始问题

基于 Task-80 数据分析，发现20条错误判定为"是"的数据，主要问题包括：

| 问题类型 | 数量 | 占比 | 说明 |
|---------|------|------|------|
| 匿名报警 | 4条 | 20% | "匿名"虽有身份证号但无真实姓名 |
| 拒绝提供/无法登记 | 2条 | 10% | 明确拒绝登记小孩等信息 |
| 研判无果 | 1条 | 5% | 系统查询无果 |
| 一方信息缺失 | 8条 | 40% | 只有一方身份证，对方信息缺失 |
| 隐含当事人缺失 | 3条 | 15% | 小孩、网友等未提供身份证 |
| 其他格式错误 | 2条 | 10% | 护照号格式错误等 |

### 2.2 根本原因

1. **规则理解偏差**
   - 系统将"所有涉警当事人"理解为"至少一方涉警当事人"
   - 导致大量只有一方信息的案件被错误判定为"是"

2. **关键词检测失效**
   - "匿名"、"拒绝登记"等关键词只触发警告，未推翻结论
   - 后置验证器缺乏强制推翻机制

3. **隐含当事人识别不足**
   - 无法识别小孩、中介、网友等隐含当事人
   - 导致这些人员的信息缺失未被检测

4. **LLM理解限制**
   - DeepSeek 70B 对复杂规则的理解能力有限
   - 需要更强的规则预处理和后置验证

## 三、修复方案

### 3.1 修复架构

采用 **RULE_THEN_LLM + POST_VALIDATION** 三层防护架构：

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 规则预处理（RULE_THEN_LLM）                         │
│  - 结构化提取所有涉警当事人                                    │
│  - 强制检测信息缺失关键词                                      │
│  - 严格验证身份证号/护照号格式                                 │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 2: LLM综合分析（DeepSeek 70B）                        │
│  - 接收预处理的结构化数据                                      │
│  - 严格遵循"所有涉警当事人"逻辑                                │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 3: 后置规则验证（PostProcessValidator）                │
│  - 强制推翻机制：发现关键词直接判"否"                           │
│  - 当事人完整性二次检查                                        │
│  - 身份证号/护照号格式最终验证                                 │
└─────────────────────────────────────────────────────────────┘
```

```java
private List<Party> extractImpliedParties(String text) {
    List<Party> parties = new ArrayList<>();

    // 1. 小孩检测
    if (text.contains("小孩") || text.contains("儿童")) {
        // 检查是否有孩子的身份证号
        Pattern childIdPattern = Pattern.compile(
            "(?:小孩|儿童)[^（]*（[^)]*?(?:身份证)[^：]*：[^\\d]*)(\\d{17}[\\dXx])");
        if (!childIdPattern.matcher(text).find()) {
            parties.add(Party.builder()
                .name("小孩/儿童")
                .type(PartyType.VICTIM)
                .isGeneric(true)
                .impliedType("小孩")
                .reason("涉及小孩/儿童但未提取到身份证号")
                .build());
        }
    }

    // 2-6. 其他隐含当事人检测...
    return parties;
}
```

#### 强制推翻机制
```java
private ValidationResult validatePositiveResultEnhanced(ValidationContext context) {
    String text = context.getText();

    // 0.1 检查匿名报警
    if (text.contains("匿名") || text.contains("匿名报警")) {
        return ValidationResult.invalid(
            "检测到[匿名]关键词，匿名报警无完整身份信息，强制判定为[否]");
    }

    // 0.2 检查强制信息缺失关键词
    for (String keyword : FORCE_OVERRIDE_KEYWORDS) {
        if (text.contains(keyword)) {
            return ValidationResult.invalid(
                String.format("检测到信息缺失关键词[%s]，强制判定为[否]", keyword));
        }
    }

}
```

## 六、DeepSeek 70B 限制应对

### 6.1 认知到的限制

| 限制 | 影响 | 应对策略 |
|------|------|---------|
| 上下文理解能力有限 | 复杂规则理解偏差 | 简化提示词，明确检查清单 |
| 逻辑推理能力不足 | "所有涉警当事人"理解偏差 | 规则预处理 + 后置验证 |
| 输出一致性不够 | 同样输入可能不同输出 | 设置低温度参数（0.1-0.3） |

### 6.2 架构设计考虑

1. **规则优先原则**
   - 确定性逻辑 > 概率性逻辑
   - 代码提取 > LLM理解

2. **强制推翻机制**
   - 关键问题检测到后直接推翻LLM结论
   - 不依赖LLM的二次修正

3. **简化提示词**
   - 给出明确的检查步骤
   - 避免开放性问题

4. **结构化提取**
   - 用代码提取结构化数据
   - 减少LLM理解负担

- ✅ 需要高准确率的数据标注场景
