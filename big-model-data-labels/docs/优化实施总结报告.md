### 3.2 L3: 提示词强化

**文件**: `BuiltinGlobalLabelsInitializer.java`
**方法**: `buildPolicePersonnelLabelDescription()`

**关键增强**:

1. **明确18位验证要求**:
   ```
   ✓ 身份证号必须严格18位（前17位为数字，第18位为数字或X）
   ✗ 身份证号非18位（17位、19位、15位等均不合格）
   ```

2. **扩展涉警当事人识别范围**:
   ```
   • 特殊注意（隐含当事人）：
     - 小孩（如被咬伤、被打、被推搡等）
     - 中介（如中介纠纷、中介介绍工作等）
     - 收费方/店家/物业（如停车费纠纷、购物纠纷等）
   ```

3. **强调信息缺失关键词判定**:
   ```
   ✗ 出现"拒绝提供"、"未登记"、"研判无果"等明确表示信息缺失的关键词
   ```

4. **新增判定步骤**:
   ```
   1. 识别所有涉警当事人（包括明确标注的和从纠纷上下文推断的）
   2. 对每个当事人提取身份证号/护照号
   3. 严格验证身份证号格式（必须18位）
   4. 检查是否存在信息缺失关键词
   5. 所有人信息完整 → "是"；任一人信息缺失 → "否"
   ```

---

### 3.3 L3: 二次强化模板优化

**文件**: `BuiltinGlobalLabelsInitializer.java`
**方法**: `buildPolicePersonnelEnhancementTemplate()`

**新增严格检查项**:

1. **[1] 身份证号格式严格验证**:
   ```
   • 19位、17位、15位等任何非18位号码均为格式错误
   • 即使推理中说明[通过校验]，长度错误仍应判定为不合格
   • 示例错误：340404199711180211X（19位）、34282519610314710（17位）
   ```

2. **[2] 涉警当事人完整性检查**:
   ```
   • 不仅识别明确标注的[报警人]、[当事人]
   • 还需从纠纷上下文中识别隐含当事人
   ```

3. **[3] 信息缺失关键词识别**:
   ```
   • 以下关键词直接表示信息不完整，应判定为[否]
   • 即使只出现一次，也表明存在信息缺失当事人
   ```

4. **[4] 豁免条件正确使用**:
   ```
   • [撤警]、[无效警情]仅当明确表示无案件、无纠纷时可直接判定为[是]
   • 存在纠纷但协商解决的，仍需检查所有涉警当事人信息
   ```

```java
/**
 * 增强的触发条件判断，支持强制触发
 */
public boolean shouldTrigger(int initialConfidence, String validationResult) {
    // 基础触发条件
    if (shouldTrigger(initialConfidence)) {
        return true;
    }

    // ✅ 新增：强制触发条件
    if (validationResult != null) {
        // 格式错误强制触发
        if (containsAny(validationResult, "格式错误", "长度错误", "17位", "19位")) {
            return true;
        }

        // 信息缺失强制触发
        if (containsAny(validationResult, "拒绝", "未提供", "研判无果")) {
            return true;
        }
    }

    return false;
}
```

**变更**: 集成后置验证器

```java
// 新增依赖
private final PostProcessValidator postProcessValidator;

// 新增方法
private EnhancementResult performPostValidation(
    String result, int confidence,
    Map<String, Object> rowData,
    List<ExtractedNumber> extractedNumbers,
    String validationResult) {

    // 执行后置验证
    PostProcessValidator.ValidationResult validation =
        postProcessValidator.validate(result, rowData, extractedNumbers);

    if (!validation.isValid()) {
        // 修正结果为"否"
        return EnhancementResult.builder()
            .finalResult("否")
            .finalConfidence(Math.min(confidence, 50))
            .wasAdjusted(true)
            .build();
    }

}
```

## 四、新增工具类详解

### 4.1 IdCardLengthValidator

**位置**: `com.datalabeling.util.IdCardLengthValidator`

**功能**:
- 严格验证18位身份证号格式
- 支持快速验证和详细验证
- 支持从文本中批量提取并验证

**核心方法**:
```java
// 严格18位验证
public static ValidationResult validate(String idNumber, boolean allowLegacy)

// 从文本中提取并验证
public static ExtractResult extractAndValidate(String text, boolean allowLegacy)

// 快速验证
public static boolean quickValidate(String idNumber)
```

**使用示例**:
```java
// 验证单个身份证号
ValidationResult result = IdCardLengthValidator.validate("340404199711180211X", false);
if (!result.isValid()) {
    System.out.println("格式错误: " + result.getReason());
}

// 从文本中提取并验证
ExtractResult extractResult = IdCardLengthValidator.extractAndValidate(text, false);
if (extractResult.hasInvalid()) {
    System.out.println("发现无效身份证号: " + extractResult.getInvalidReasons());
}
```

### 4.2 PostProcessValidator

**位置**: `com.datalabeling.util.PostProcessValidator`

**功能**:
- 对LLM判定结果进行规则层面的最终验证
- 防止LLM误判（特别是格式错误和信息缺失）
- 提供详细的验证报告

**核心方法**:
```java
public ValidationResult validate(
    String result,              // LLM判定结果
    Map<String, Object> rowData, // 原始数据
    List<ExtractedNumber> extractedNumbers // 提取的号码
)
```

**验证规则**:
1. 身份证号格式检查（严格18位）
2. 信息缺失关键词检测
3. 隐含当事人识别
4. 当事人完整性验证

**验证级别**:
- `SUCCESS`: 验证通过
- `WARNING`: 可能误判，需人工复核
- `ERROR`: 确定误判，已自动修正
