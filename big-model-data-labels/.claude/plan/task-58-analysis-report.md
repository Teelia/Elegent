## 三、模型行为分析

### 提取策略

代码 `NumberEvidenceExtractor.java` 使用的正则表达式：

```java
private static final Pattern DIGITS_14_22 = Pattern.compile("(?<!\\d)\\d{14,22}(?![\\dXx])");
```

**关键特征**：
- `(?<!\d)` - 负向后顾：前面不能是数字
- `(?![\dXx])` - 负向前瞻：后面不能是数字或X

### 为什么没有提取案件编号中的子串？

对于案件编号 `A3404226100002025120115`：
- 前18位 `340422610000202512` 前面是 `A`（不是数字）✓
- 后面是 `0115`（是数字）✗
- **因此不会被提取**

## 四、这是正确的设计决策

### 为什么不提取长数字串的子串？

1. **真实身份证号应该是独立的18位数字**
   - 不会是更长数字串的一部分
   - 会有明确的边界（空格、标点、换行等）

2. **避免大量误报**
   - 案件编号：A3404226100002025120115 (23位)
   - 银行流水号：20260126123456789012 (20位)
   - 订单号：202601263404221997111802115 (24位)

3. **业务口径清晰**
   - "错误身份证号"应定义为：**独立的、格式完整的、但校验位错误的18位数字**
   - 不应包含：更长数字串的子串

### 如果提取子串会怎样？

假设修改策略，提取所有18位数字：

```
输入: "案件编号：A3404226100002025120115"
提取: 340422610000202512 (无效)
     3404226100002025120115 (23位，长度错误)
     6100002025120115 (从第7位开始，不符合身份证结构)
```

问题：
- 产生大量误报
- 需要额外的上下文过滤逻辑
- 增加复杂度和不确定性

## 六、结论与建议

### 主要发现

1. **测试数据特殊性**
   - 唯一的"无效身份证号"实际上是案件编号的子串
   - 不符合"独立的18位数字"这一业务口径

2. **模型行为正确**
   - 使用负向前瞻避免提取长数字串的子串
   - 这是合理的设计，避免大量误报

3. **准确率83.3%的真相**
   - 5行完全正确
   - 1行"漏报"实际上是正确的设计决策

### 业务口径说明

**"错误身份证号"的正确定义**：
- ✓ 独立的18位数字（前后有明确的文本边界）
- ✓ 符合身份证结构（区域码、出生日期）
- ✗ 校验位错误或格式错误

**不包含**：
- ✗ 案件编号、订单号、流水号等长数字串的子串
- ✗ 即使这些子串恰好符合身份证格式

### 建议

#### 1. 保持现有提取策略 ✓
- 负向前瞻 `(?![\dXx])` 是正确的
- 避免把长数字串拆成多个候选

- **添加**: 真正的独立无效身份证号，例如：

  ```
  期望: 识别为无效身份证号
  ```

#### 3. 文档完善
- 在标签说明中明确：仅识别独立的18位数字
- 说明不会提取长数字串的子串

#### 4. 如果需要识别案件编号错误
- 应创建专门的标签："错误案件编号"
