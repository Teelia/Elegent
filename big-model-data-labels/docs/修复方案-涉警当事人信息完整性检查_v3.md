# 涉警当事人信息完整性检查 v3 - 全面修复方案

## 一、问题总结

基于 Task-80 的错误分析，发现以下核心问题：

### 1.1 规则预处理层问题
- ❌ "匿名"关键词未包含在负面关键词列表
- ❌ 隐含当事人（小孩、中介、收费方）未充分识别
- ❌ 护照号格式验证不足（学号被误识别）

### 1.2 后置验证层问题
- ❌ 信息缺失关键词仅触发"警告"，未强制推翻"是"的结论
- ❌ 当事人完整性检查可能未生效

### 1.3 LLM分析层问题
- ❌ DeepSeek 70B理解"所有涉警当事人"存在偏差
- ❌ 可能将"至少一方完整"理解为"满足要求"

## 二、修复架构设计

### 2.1 三层防护体系

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 规则预处理（RULE_THEN_LLM）                         │
│  - 结构化提取所有涉警当事人                                    │
│  - 强制检测信息缺失关键词                                      │
│  - 严格验证身份证号/护照号格式                                 │
│  - 输出：结构化提取结果 + 强制判定建议                          │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 2: LLM综合分析（DeepSeek 70B）                        │
│  - 接收预处理的结构化数据                                      │
│  - 严格遵循"所有涉警当事人"逻辑                                │
│  - 输出：初步判断 + 置信度 + 推理                              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  Layer 3: 后置规则验证（PostProcessValidator）                │
│  - 强制推翻机制：发现关键词直接判"否"                           │
│  - 当事人完整性二次检查                                        │
│  - 身份证号/护照号格式最终验证                                 │
│  - 输出：最终结果 + 验证说明                                    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 关键设计原则

1. **规则优先于LLM**：确定性逻辑（规则）优先于概率性逻辑（LLM）
2. **强制推翻机制**：关键问题检测到后，直接推翻LLM结论
3. **结构化提取**：用代码结构化提取当事人，而非依赖LLM理解
4. **逐步细化**：从粗粒度到细粒度，层层过滤

### 3.1 【P0-核心】PartyExtractor增强 - 隐含当事人识别

**文件**: `PartyExtractor.java`

**问题**: 当前只提取明确标注的"报警人"、"对方当事人"，无法识别隐含的涉警人员。

**修复方案**:

```java
/**
 * 新增：隐含当事人提取器
 */
private List<Party> extractImpliedParties(String text) {
    List<Party> parties = new ArrayList<>();

    // 1. 小孩检测（被咬伤、被打、被推搡等）
    if (text.contains("小孩") || text.contains("儿童") || text.contains("孩子")) {
        // 检查是否有孩子的身份证号
        Pattern childIdPattern = Pattern.compile(
            "(?:小孩|儿童|孩子)[^（]*（(?:[^）]*?(?:身份证|身份码)[^：]*：[^\\d]*)(\\d{17}[\\dXx])");
        Matcher m = childIdPattern.matcher(text);
        if (!m.find()) {
            // 没有找到孩子的身份证号，添加缺失
            parties.add(Party.builder()
                .name("小孩/儿童")
                .type(PartyType.VICTIM)
                .isGeneric(true)
                .impliedType("小孩")
                .reason("涉及小孩但未提取到身份证号")
                .build());
        }
    }

    // 2. 中介检测（中介纠纷、中介介绍等）
    if (text.contains("中介") && !text.matches(".*中介[：:\\s]*[\\d]{11}.*")) {
        parties.add(Party.builder()
            .name("中介")
            .type(PartyType.OTHER)
            .isGeneric(true)
            .impliedType("中介")
            .reason("提及中介但未提取到身份信息")
            .build());
    }

    // 3. 收费方/店家/物业（停车费、购物纠纷等）
    List<String> merchantKeywords = Arrays.asList("收费方", "店家", "物业", "商家", "店主");
    for (String keyword : merchantKeywords) {
        if (text.contains(keyword)) {
            // 检查是否有该方的身份证号
            Pattern pattern = Pattern.compile(keyword + "[^（]*（(?:身份证)...");
            // 简化：如果提及但后续未跟身份证号，则添加缺失
            parties.add(Party.builder()
                .name(keyword)
                .type(PartyType.OTHER)
                .isGeneric(true)
                .impliedType("收费方")
                .reason("提及" + keyword + "但未提取到身份信息")
                .build());
            break; // 避免重复
        }
    }

    // 4. 网友/网友见面（诈骗、纠纷等）
    if ((text.contains("网友") || text.contains("微信网友") || text.contains("抖音网友"))
        && !text.matches(".*网友[^（]*（[^）]*?身份证[^：]*：[^）]*）.*")) {
        parties.add(Party.builder()
            .name("网友")
            .type(PartyType.OTHER)
            .isGeneric(true)
            .impliedType("网友")
            .reason("提及网友但未提取到身份信息")
            .build());
    }

    // 5. 犯罪嫌疑人/侵权人（未明确身份）
    if (text.contains("犯罪嫌疑人") || text.contains("嫌疑人")) {
        // 检查是否提供了身份信息
        Pattern suspectPattern = Pattern.compile(
            "(?:犯罪嫌疑人|嫌疑人)[^（]*（(?:[^）]*?(?:身份证|身份码)[^：]*：[^\\d]*)(\\d{17}[\\dXx]))");
        if (!suspectPattern.matcher(text).find()) {
            parties.add(Party.builder()
                .name("犯罪嫌疑人")
                .type(PartyType.SUSPECT)
                .isGeneric(true)
                .impliedType("犯罪嫌疑人")
                .reason("提及犯罪嫌疑人但未提取到身份信息")
                .build());
        }
    }

    return parties;
}
```

- ✅ 检测"小孩/儿童"并提取身份证号
- ✅ 检测"中介"身份信息
- ✅ 检测"收费方/店家/物业"
- ✅ 检测"网友"身份信息
- ✅ 检测"犯罪嫌疑人"身份信息

### 3.2 【P0-核心】PostProcessValidator强化 - 强制推翻机制

**文件**: `PostProcessValidator.java`

**问题**: 信息缺失关键词只触发警告，不推翻结论。

**修复方案**:

```java
/**
 * 增强的信息缺失关键词列表（包含"匿名"）
 */
private static final List<String> MISSING_INFO_KEYWORDS_FORCE = Arrays.asList(
    "匿名", "匿名报警", "匿名报警人",
    "拒绝", "拒绝登记", "拒绝提供", "拒不透露",
    "未提供", "未登记", "无法", "无法登记", "无法提供",
    "研判无果", "查询无果", "系统查询无果", "无果",
    "不详", "记不清", "身份信息不详", "未获取到",
    "无法核实", "无法核实身份"
);

/**
 * 增强的"是"判断验证
 */
private ValidationResult validatePositiveResultEnhanced(ValidationContext context) {
    List<String> errors = new ArrayList<>();
    List<String> warnings = new ArrayList<>();

    String text = context.getText();

    // ========== 0. 强制推翻检查（最高优先级）==========
    // 0.1 检查匿名报警
    if (text.contains("匿名") || text.contains("匿名报警") || text.contains("匿名报警人")) {
        return ValidationResult.invalid(
            "检测到[匿名]关键词，匿名报警无完整身份信息，强制判定为[否]");
    }

    // 0.2 检查强制信息缺失关键词
    for (String keyword : MISSING_INFO_KEYWORDS_FORCE) {
        if (text.contains(keyword)) {
            // 检查是否有足够的身份证号来抵消这个关键词
            // 策略：如果提及"拒绝"但未提供该人的替代信息，则强制判否
            return ValidationResult.invalid(
                String.format("检测到信息缺失关键词[%s]，强制判定为[否]", keyword));
        }
    }

    // 0.3 检查研判无果
    if (text.contains("研判无果") || text.contains("查询无果") || text.contains("系统查询无果")) {
        return ValidationResult.invalid(
            "检测到[研判无果/查询无果]，表明无法获取当事人身份信息，强制判定为[否]");
    }

    // ========== 1. 身份证号格式检查 ==========
    IdCardLengthValidator.ExtractResult idResult = context.getIdCardValidationResult();
    if (idResult.hasInvalid()) {
        List<String> invalidDetails = idResult.getInvalidDetails();
        for (String detail : invalidDetails) {
            errors.add("身份证号格式错误: " + detail);
        }
    }

    // ========== 2. 护照号格式检查 ==========
    PassportValidator.ExtractResult passportResult = context.getPassportValidationResult();
    if (passportResult.hasSuspiciousPassports()) {
        errors.add("护照号格式不符合标准: " + passportResult.getInvalidReasons());
    }

    // ========== 3. 当事人完整性检查（核心）==========
    PartyExtractor.CompletenessResult completenessResult =
            partyExtractor.checkCompleteness(context.getParties());

    if (!completenessResult.isComplete()) {
        // 核心问题：当事人信息不完整
        errors.add(completenessResult.getMessage());
    }

    // ========== 生成结果 ==========
    if (!errors.isEmpty()) {
        return ValidationResult.invalid(
            String.format("判定为[是]但发现%d个问题：%s",
                errors.size(),
                String.join("; ", errors)));
    }

    if (!warnings.isEmpty()) {
        return ValidationResult.warning(
            String.format("判定为[是]，但有%d个警告：%s",
                warnings.size(),
                String.join("; ", warnings)));
    }

    return ValidationResult.valid("规则验证通过，判定结果正确");
}
```

**关键改进**:
- ✅ "匿名"直接强制推翻
- ✅ "拒绝登记"直接强制推翻
- ✅ "研判无果"直接强制推翻
- ✅ 不再仅触发警告

### 3.6 【P2-优化】护照号格式验证增强

**文件**: `PassportValidator.java`

**问题**: 学号"BL19004002"被误认为护照号。

**修复方案**:

```java
/**
 * 增强的护照号验证
 */
public ExtractResult extractAndValidate(String text) {
    List<String> passports = new ArrayList<>();
    List<String> invalidReasons = new ArrayList<>();

    // 中国护照号格式：E/G + 8位数字（共9位）
    Pattern cnPassportPattern = Pattern.compile("\\b([EG])[0-9]{8}\\b");
    Matcher cnMatcher = cnPassportPattern.matcher(text);
    while (cnMatcher.find()) {
        passports.add(cnMatcher.group());
    }

    // 外国护照号格式（各国不同，通常8-10位字母数字组合）
    // 排除学号格式（如 BL19004002 是学号，不是护照号）
    Pattern foreignPassportPattern = Pattern.compile(
        "\\b(?![A-Z]{2}\\d{8}\\b)[A-Z]{1,2}[0-9A-Z]{6,9}\\b");
    Matcher foreignMatcher = foreignPassportPattern.matcher(text);
    while (foreignMatcher.find()) {
        String candidate = foreignMatcher.group();
        // 排除明显的学号格式
        if (candidate.matches("^[A-Z]{2}\\d{8}$")) {
            // 可能是学号，添加警告
            invalidReasons.add("可疑护照号（可能是学号）: " + candidate);
            continue;
        }
        passports.add(candidate);
    }

    // 检测可疑护照号
    boolean hasSuspicious = !invalidReasons.isEmpty();

    return new ExtractResult(passports, invalidReasons, hasSuspicious);
}
```

## 七、DeepSeek 70B 限制应对

### 7.1 认知到的限制
- 上下文理解能力不如 GPT-4
- 复杂逻辑推理可能出错
- 对"所有涉警当事人"理解可能偏差

### 7.2 应对策略
1. **规则优先**: 用确定性逻辑替代概率性逻辑
2. **强制推翻**: 关键问题检测到后直接推翻LLM结论
3. **结构化提取**: 用代码提取结构化数据，而非依赖LLM理解
4. **简化提示词**: 给出明确的检查清单，而非开放性问题

### 7.3 为什么选择 RULE_THEN_LLM
- **规则预处理**: 提取结构化数据，减少LLM理解负担
- **LLM分析**: 基于结构化数据进行简单判断
- **后置验证**: 规则层面的最终保障

这种架构能够在 DeepSeek 70B 限制下达到最佳效果。
