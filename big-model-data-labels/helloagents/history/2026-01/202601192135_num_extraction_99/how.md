# 技术设计: 号码类标签提取与分析链路 99% 准确改造

## 技术方案

### 核心技术
- Java 8 规则引擎（正则 + 校验算法 + 冲突仲裁）
- 结构化证据 JSON（脱敏、可审计）
- DeepSeek 70B（仅做分析解释/少量歧义补充）
-（可选）二次强化：一致性校验为主，LLM 复核为辅

### 现状问题归因（来自 Task43 回归）
现有“智能识别器”存在典型的冲突吞噬：
- 按类型提取候选时，**先记录范围再验证**，导致银行卡类型先占用 16-19 位数字范围，即便银行卡校验失败也占用。
- 身份证类型后续抽取被“范围重叠”跳过，从而出现“未找到身份证号”的系统性漏检。

### 目标架构（你期望的流程落地）

```mermaid
flowchart TD
    A[原始数据] --> B[规则预处理: 归一化+候选抽取+分类校验]
    B --> C[数据建模: 证据JSON(脱敏)]
    C --> D[构建LLM提示词(证据约束)]
    D --> E[DeepSeek70B 分析]
    E --> F[一致性校验/二次强化(可选)]
    F --> G[输出结论/结构化提取结果/needs_review]
```

## 架构设计

### 组件拆分（建议）
1. **NumberEvidenceExtractor（新）**
   - 输入：原始文本（可由 focus_columns 拼接而来）
   - 输出：证据模型 JSON（候选列表 + 校验 + 冲突仲裁 + 派生字段）
2. **PromptBuilder（改造）**
   - 输入：标签描述 + 证据 JSON（提示词版脱敏）+（可选）最小原文窗口
   - 输出：强约束提示词：LLM 只能引用证据ID/候选值
3. **RuleConsistencyValidator（新，可选）**
   - 校验 LLM 输出是否引用了证据、结论是否与派生字段一致
   - 不一致 → needs_review，并记录原因

## 实现要点

### 1) 归一化与候选抽取
- 支持从含分隔符文本中提取候选（空格/短横线/中文逗号/顿号等）。
- 建议策略：
  - 扫描“数字与X/x及分隔符”的片段，归一化为纯数字或数字+X
  - 保留 span（原文位置），便于审计与 UI 高亮

### 2) 分类与校验（规则为 SSOT）

#### 身份证（可验证项）
- 18 位：结构校验 + 出生日期合法 + 校验位算法（ISO 7064）通过
- 15 位：结构校验（是否视为“错误身份证”由业务决定）
- 非18位“疑似身份证”：
  - 具备身份证结构特征（地区码首位1-6 + 生日位 YYYYMMDD 可解析）
  - 长度为 16/17/19/20/21/22 等时判为 `ID_INVALID_LENGTH`

#### 银行卡（可验证项）
- 16-19 位：Luhn 校验通过（强信号）
- BIN 校验可选（BIN 库不完整，不应作为唯一判断）
- 与身份证冲突时（例如 Luhn 通过但同时具身份证结构）：
  - 优先按身份证结构裁决，并记录 conflict（避免吞噬 Task43）

#### 手机号
- 11 位且满足 `^1[3-9]\\d{9}$`，并做边界检查（避免从更长串中截出 11 位子串）。

### 3) 冲突仲裁（关键）
必须避免“先占范围再验证”的策略。推荐：
- **候选抽取阶段不做跨类型排他占用**（或只对“已验证通过/高置信”结果占用）。
- 仲裁阶段按“强校验通过 > 结构强特征 > 弱特征”排序确定最终类型。
- 对冲突保留可追溯信息（conflicts），供 needs_review 或审计。

### 4) LLM 提示词约束（保证稳定性）
建议把 LLM 的职责限定为：
- 基于证据 JSON 生成业务解释/结论摘要
- 不能生成证据外的号码
- 必须输出可校验结构（JSON）并引用 `numbers[].id`

### 5) 二次强化（可选但推荐“规则校验优先”）
二次强化不建议采用“同模型审同模型”作为主手段，建议：
- **一致性校验（硬规则）**：证据引用、结论与 derived 字段一致
- **LLM 复核（软补充）**：仅在冲突/歧义样本中启用，输出建议但不自动覆盖规则裁决

## 安全与性能
- **安全:**
  - 全链路脱敏：日志/提示词/导出/前端默认脱敏
  - 证据 JSON 中原文片段默认不存储，仅存 span 与指纹
- **性能:**
  - 规则预处理 O(n) 扫描 + 常量校验，单行可在毫秒级完成
  - 允许性能略慢的前提下，可对疑难样本启用 LLM 二次复核

## 测试与部署
- **回归评估:**
  - 固化回归集：从现有 360 行中抽样并补充边界样本，人工标注真值（存在/错误/类型）
  - 指标：precision/recall/F1 + needs_review 比例
- **单元测试:**
  - 覆盖 17/19/22 位身份证结构候选
  - 覆盖“Luhn 通过但具身份证结构”冲突样本
- **灰度策略:**
  - 新旧链路并行输出（对比日志/指标），稳定后切换为新链路

## 架构决策 ADR

### ADR-001: 号码类任务以规则证据为SSOT，LLM只做分析与解释
**上下文:** 号码类结论需要可验证与可审计；LLM 存在幻觉与不稳定，不适合作为最终裁判。  
**决策:** 引入证据 JSON 作为 SSOT；LLM 输出必须引用证据ID；最终裁决以规则一致性校验为准。  
**替代方案:** LLM 直接抽取并裁决 → 拒绝原因：难以稳定达到 99% 且不可审计。  
**影响:** 增加规则建模与一致性校验模块，但显著提升稳定性与可度量性。
