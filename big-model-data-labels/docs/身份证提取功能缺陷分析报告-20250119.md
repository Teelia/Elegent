## ✅ 修复更新（2026-01-19）

已在后端引入“号码证据建模（规则侧SSOT）”并替换 Task43 的否定条件预处理路径，解决系统性漏检问题：

- **根因修正**：并非“身份证验证正则过严导致无法提取”，而是“跨类型冲突处理策略”导致候选被吞掉：银行卡类型先匹配并占用范围/去重，即使校验失败也会阻塞后续身份证候选提取。
- **修复方式**：否定条件任务改用 `NumberEvidenceExtractor` 做规则建模与裁决（不再依赖 SmartCardIdentifier 的范围占用策略），并新增回归测试覆盖该数据集。
- **口径补充**：15位（一代身份证）视为正确号码，**不纳入**“不满足18位的错误身份证号”输出。

对应实现与回归测试见：
- `backend/src/main/java/com/datalabeling/service/extraction/NumberEvidenceExtractor.java`
- `backend/src/main/java/com/datalabeling/service/extraction/NegativeConditionPreprocessor.java`

## 3. 根本原因分析

### 3.1 设计目标错位

| 组件 | 设计目标 | 当前任务需求 | 冲突 |
|------|----------|--------------|------|
| **SmartCardIdentifier** | 识别**有效的、格式正确**的卡号 | 提取**所有可能的**候选号码（包括错误的） | ❌ 矛盾 |

**核心矛盾**：
- `SmartCardIdentifier` 的目标是"过滤掉格式错误的号码"
- 当前任务需要"找到格式错误的号码"

### 3.2 代码流程问题

#### 当前代码流程（NegativeConditionPreprocessor.java）

**步骤2的问题**：`getCardsByType(CardType.ID_CARD)` 返回什么？

这个方法返回的是 `cardsByType` 中的号码，这些号码是**通过格式验证**的。

#### 格式验证规则（CardType.java）


**验证要求**：

## 5. 解决方案建议

### 5.1 问题确认

经过深入分析，问题的核心是：

**`SmartCardIdentifier` 在验证阶段会过滤掉格式不正确的号码，**
**但对于"提取错误的身份证号"任务，我们需要的是保留所有候选号码。**

### 5.2 解决方案对比

#### 方案A：直接使用宽松匹配（推荐）⭐

**实现位置**：`NegativeConditionPreprocessor.processInvalidIdCardByLength()`

**核心思路**：
```java
private PreprocessResult processInvalidIdCardByLength(...) {
    String dataString = convertDataToString(rowData);

    // 不使用 SmartCardIdentifier，直接用宽松正则
    Pattern pattern = Pattern.compile("\\d{14,22}");
    Matcher matcher = pattern.matcher(dataString);

    Set<String> allCandidates = new HashSet<>();
    List<String> invalidIds = new ArrayList<>();

    while (matcher.find()) {
        String number = matcher.group();

        // 过滤明显是银行卡号的号码
        if (isLikelyBankCardNumber(number)) {
            continue;
        }

        allCandidates.add(number);

        // 筛选长度不等于18位的
        if (number.length() != 18) {
            invalidIds.add(number);
        }
    }

    // ... 返回结果
}
```

**优点**：
- ✅ 简单直接，逻辑清晰
- ✅ 不改动现有 `SmartCardIdentifier`
- ✅ 专门针对"错误号码"任务优化
- ✅ 符合 YAGNI 和 KISS 原则

**缺点**：
- ⚠️ 需要添加银行卡号过滤逻辑
- ⚠️ 可能误识别其他长数字串

#### 方案B：在 SmartCardIdentifier 中添加"宽松模式"

**实现位置**：`SmartCardIdentifier` 新增方法

**核心思路**：
```java
public class SmartCardIdentifier {
    // 新增方法：返回所有候选号码（不验证格式）
    public List<String> extractAllCandidates(String text, CardType type) {
        List<String> results = new ArrayList<>();
        String regex = "\\d{" + type.getMinLength() + "," + type.getMaxLength() + "}";
        Pattern pattern = Pattern.compile(regex);
        Matcher matcher = pattern.matcher(text);

        while (matcher.find()) {
            results.add(matcher.group());
        }

        return results;
    }
}
```

**优点**：
- ✅ 扩展性强，其他任务也可以使用
- ✅ 保持代码结构统一

**缺点**：
- ❌ 修改核心类，可能影响其他功能
- ❌ 增加接口复杂度
- ❌ 违反"当前问题不过度设计"原则

#### 方案C：修改验证逻辑

**实现位置**：`CardType.ID_CARD` 的验证正则

**核心思路**：
```java
// 修改前（严格验证）：
ID_CARD("^[1-9]\\d{5}(19|20)\\d{2}(0[1-9]|1[0-2])\\d{3}[\\dXx]$|^\\d{15}$")

// 修改后（宽松验证）：
ID_CARD("^\\d{14,22}$")  // 只验证长度范围
```

**优点**：
- ✅ 最小化代码改动

**缺点**：
- ❌ 破坏现有功能，影响所有使用 ID_CARD 的地方
- ❌ 失去格式验证能力
- ❌ 不可接受

### 5.3 推荐方案详细设计

#### 方案A 实现细节

```java
/**
 * 处理"不满足18位的身份证号"任务（修复版）
 *
 * 优化策略：
 * 1. 直接使用宽松正则提取所有候选号码
 * 2. 基于启发式规则过滤明显非身份证的号码
 * 3. 筛选长度不等于18位的号码
 */
private PreprocessResult processInvalidIdCardByLength(...) {
    String dataString = convertDataToString(rowData);
    if (dataString == null || dataString.isEmpty()) {
        return PreprocessResult.empty("数据为空");
    }

    // 步骤1：提取所有14-22位的数字序列
    Pattern pattern = Pattern.compile("\\d{14,22}");
    Matcher matcher = pattern.matcher(dataString);

    Set<String> allCandidates = new HashSet<>();
    Set<String> likelyBankCards = new HashSet<>();
    Set<String> invalidIds = new HashSet<>();

    while (matcher.find()) {
        String number = matcher.group();

        // 去重
        if (!allCandidates.add(number)) {
            continue;
        }

        // 步骤2：过滤明显是银行卡号的号码
        if (isLikelyBankCardNumber(number)) {
            likelyBankCards.add(number);
            continue;
        }

        // 步骤3：筛选长度不等于18位的
        if (number.length() != 18) {
            invalidIds.add(number);
        }
    }

    // 调试日志
    log.debug("[宽松预处理器] 总候选: {}, 银行卡: {}, 错误身份证: {}",
        allCandidates.size(), likelyBankCards.size(), invalidIds.size());

    if (invalidIds.isEmpty()) {
        String reason = allCandidates.isEmpty()
            ? "未找到可能的身份证号"
            : "所有号码都是18位";
        return PreprocessResult.empty(reason);
    }

    String resultText = String.join(", ", invalidIds);
    String reasoning = String.format(
        "从数据中找到 %d 个候选号码，过滤 %d 个银行卡号，其中 %d 个不满足18位要求：%s [规则预处理: 宽松模式]",
        allCandidates.size(), likelyBankCards.size(), invalidIds.size(), resultText
    );

    return PreprocessResult.success(resultText, 90, reasoning);
}

/**
 * 判断一个数字串是否可能是银行卡号
 *
 * 银行卡特征：
 * - 中国银联卡以62开头
 * - 常见BIN码：621-629（各大银行）
 * - 长度通常是16-19位
 */
private boolean isLikelyBankCardNumber(String number) {
    if (number == null || number.length() < 16) {
        return false;
    }

    // 19位且以621-629开头，很可能是银行卡
    if (number.length() == 19) {
        if (number.matches("^62[1-9]\\d+")) {
            return true;
        }
    }

    // 16-18位且以621-629开头，也可能是银行卡
    if (number.length() >= 16 && number.length() <= 18) {
        if (number.matches("^62[1-9]\\d+")) {
            // 进一步检查：如果中间有连续的0，很可能是银行卡
            if (number.contains("000") || number.contains("00")) {
                return true;
            }
        }
    }

    return false;
}
```
