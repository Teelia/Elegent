# 技术设计: 全局标签改为系统范围可见

## 技术方案

### 核心技术
- 后端：Spring Boot + Spring Data JPA
- 数据：MySQL（按现有项目约定）

### 实现要点
1. **新增 Repository 查询**：提供“不按 user_id 过滤”的全局标签查询，仍保持：
   - `scope='global'`
   - `isActive=true`
   - `name` 维度仅取最新 `version`
2. **调整 Service 路由**：当 `scope=global` 时改用“系统范围全局查询”。
3. **权限与审计**：接口仍要求登录；审计保留当前用户上下文，但数据来源不再绑定 userId。
4. **文档同步**：更新 `helloagents/wiki/api.md`，明确该接口的系统范围语义。

## API设计

### [GET] /api/labels/active?scope=global
- **变更前:** 返回当前登录用户的全局标签（激活 + 最新版本）
- **变更后:** 返回系统范围内全部全局标签（激活 + 最新版本）

## 安全与性能
- **安全:** 该方案会扩大可见性范围；实施时需要做一次权限/产品语义复核（是否允许普通用户创建全局标签）。
- **性能:** 查询条件更宽，标签量大时需依赖索引（`scope`、`is_active`、`user_id,name,version` 等）和分页策略；本接口当前为“全部返回”，后续如量大建议引入分页或缓存。

## 测试与部署
- **测试:** 增加至少一个后端集成测试用例：
  - 构造两个不同 user 的 `scope=global` 标签，验证普通用户调用 `scope=global` 能同时返回两者的最新激活版本。
  - 校验 `scope=dataset` 行为未回归。
- **部署:** 无 schema 变更；常规发布即可。

## 架构决策 ADR

### ADR-001: 全局标签的“系统范围可见”实现方式
**上下文:** 现有实现按当前用户过滤全局标签，无法满足“系统内置全局标签库”诉求。  
**决策:** 先采用最小变更：`scope=global` 查询不再按 userId 过滤，直接返回系统范围全局标签（激活 + 最新版本）。  
**理由:** 改动面小、落地快、便于验证；后续可在不破坏接口路径的前提下演进为“内置标签”模型。  
**替代方案:** “admin维护的global=内置标签”或新增 `is_builtin` 字段 → 当前被拒绝原因：需要额外权限策略/数据模型/迁移与运维流程。  
**影响:** 可见性扩大，存在“普通用户创建的global对他人可见”的风险，需要产品策略后续收敛。
