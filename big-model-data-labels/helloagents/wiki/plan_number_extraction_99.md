# 号码类标签提取与分析改造方案（目标：99%+ 可审计准确）

**适用数据**：警情/接警自由文本（以“身份证号/银行卡号/手机号”等号码类信息为主）  
**适用模型**：DeepSeek 70B（内网，OpenAI兼容接口）  
**编写日期**：2026-01-19  

---

## 1. 目标、边界与验收口径

### 1.1 目标
- 在不追求极致性能的前提下，实现“号码类标签”稳定、可回归、可审计的高准确提取与分析。
- 让“号码提取”尽量变成**可验证的确定性计算**；让 LLM 负责**语义解释/归因/上下文判断**，而不是凭空抽取号码。

### 1.2 边界（必须提前讲清，否则无法“100%准确”）
- **范围内（可做到非常高准确）**
  - 文本中是否出现身份证/银行卡/手机号等号码候选（存在性）
  - 号码是否满足格式（长度/生日段/校验位/校验算法）
  - 号码之间的冲突仲裁（同一段数字到底更像银行卡还是身份证）
- **范围外（无法仅靠文本 + LLM 达到100%）**
  - “号码是否真实存在/是否属于某人/是否已注销/是否有效证件”等真实性校验  
    需要权威库/外部系统（公安/银行/内部业务库）才能验证，LLM 也不能保证。

### 1.3 关键口径（已确认）
- **15位（一代身份证）视为正确号码**：不纳入“**不满足18位的错误身份证号**（Task43）”输出。
- **全遮挡（全部*）不计存在**：视为无用字符；仅对“包含部分数字”的遮挡片段输出 `masked` 类证据。
- “错误身份证号”建议拆成两个可验证子类：
  - `ID_INVALID_LENGTH`：非 15/18 位但具备强身份证结构特征（如地区码+生日段）。
  - `ID_INVALID_CHECKSUM`：18位结构满足但校验位不通过。
- 身份证遮挡类证据（用于标签闭环）：
  - `ID_MASKED`：长度为 15/18 且包含部分数字的遮挡身份证片段。
  - `ID_INVALID_LENGTH_MASKED`：长度不满足 15/18 且包含部分数字的遮挡身份证片段（错误长度遮挡）。

## 2. 当前项目功能与实现逻辑梳理（现状）

### 2.1 核心模块
- **frontend（Vue3）**：任务创建/标签配置/进度展示/结果复核与导出。
- **backend（Spring Boot）**：标签管理、分析任务编排、规则预处理/结构化提取、大模型调用、结果落库、导出与同步。

### 2.2 标签分析执行主链路（高层）
1. 任务服务（分析任务/异步编排）按批读取数据行。
2. 针对每个 `label` 生成输入上下文（聚焦列/原始行JSON/辅助字段等）。
3. **规则预处理（可选/优先）**：对“规则明确且可验证”的任务优先走规则侧输出结果。
4. 若规则预处理无法覆盖 → 构建提示词 → 调用 DeepSeek 70B → 解析结构化结果。
5. （可选）二次强化/校验：对不确定样本做复核或一致性检查，输出 `needs_review`。
6. 结果写入 `label_results`，供前端复核与导出。

## 3. 对“100%准确”需求的评估结论

### 3.1 结论（务实口径）
- 如果“100%准确”指：**对可验证的格式/校验规则**做到一致、可回归、无系统性漏检——可以做到接近 100%（以回归集覆盖范围为准）。
- 如果“100%准确”指：**对所有文本、所有噪声、所有缺失/遮挡/歧义**都给出确定且正确的业务结论——**不可实现**（信息论层面缺失，且 LLM 无法保证零幻觉）。

### 3.2 将 99% 变成可实现目标的关键策略
- 以“**可验证证据**”为准：规则侧输出证据（候选、校验信息、冲突信息、跨度），作为 SSOT。
- 以“**可回归指标**”为准：precision/recall/F1 + needs_review 比例，而不是只看“命中数”。
- 以“**可审计输出**”为准：LLM 必须引用证据ID，禁止输出证据外号码；不一致则自动降级/打回复核。
- 允许“**不确定**”：对缺失/遮挡/语义歧义样本输出 needs_review，比胡猜更接近“高准确”。

## 4. 推荐改造方案（与期望流程对齐）

你期望的流程：
原始数据 → 正则+代码预处理（结构化）→ JSON建模 → 构建提示词 → DeepSeek分析 → 推理+结论 →（可选）二次强化判断 → 最终结论

建议落地为“规则证据SSOT + LLM受限推理 + 一致性校验”的三段式：

### 4.1 阶段A：规则预处理与证据建模（SSOT）
- 归一化：统一全角/半角数字、分隔符、常见噪声符号。
- 候选抽取：扫描 14-22 位数字、11位手机号、17位+X 等形态。
- 分类校验（确定性）：
  - 身份证：生日段合法性、18位校验位、15位结构。
    - 近似恢复：19位纯数字串若“删除其中一位可恢复为结构+校验位均通过的18位身份证”，按错误长度身份证处理（避免生日段被插入一位导致漏检）。
  - 银行卡：Luhn 校验 +（可选）BIN/长度规则。
  - 手机号：号段正则。
- 冲突仲裁：同一候选既像银行卡又像身份证时，按“结构强度/校验结果/任务目标”裁决，避免“先占用范围导致吞噬”。
- 输出证据 JSON：包含 `numbers[]` 与 `derived`（如 `id_exists`、`id_invalid_length_count` 等），并对提示词/日志默认脱敏。

### 4.2 阶段B：LLM 受限分析（只做语义，不做凭空抽取）
- 提示词输入包含：
  - 原始文本摘要（可截断）
  - 证据列表（证据ID + 脱敏号码 + 类型 + 校验摘要）
  - 明确约束：**只能引用证据ID**，不得输出证据外号码
  - 输出 JSON schema（便于程序化解析与校验）
- DeepSeek 输出：
  - `analysis`：解释为什么选择这些证据ID
  - `result`：最终提取值/结论（引用证据ID或派生字段）

### 4.3 阶段C：一致性校验 +（可选）二次强化
优先做“规则一致性校验”，必要时再做“二次 LLM 复核”：
- 规则一致性校验（推荐必做）：
  - 输出是否只引用了存在的证据ID
  - 结论是否与证据类型/校验信息一致（如把 `BANK_CARD` 当身份证）
  - 发现不一致 → 自动标记 `needs_review` 或触发一次“纠错重问”
- 二次 LLM 复核（可选，慢但稳）：
  - 仅对 needs_review 或高风险样本触发
  - 输入：证据 + 一次结果 + 让模型判断“是否存在逻辑跳跃/证据不足”

## 5. 结合 Task43 回归集的现状洞察（不暴露明文）

基于 `analysis-task-43-results-with-reasoning.xlsx`（360行）的聚合统计（仅做结构分布，不输出号码明文）：

- 含校验通过的18位身份证候选的行：约 108/360
- 含疑似银行卡（62开头且Luhn通过）的行：约 21/360
- 15位旧身份证样本：1/360（已按口径排除，不应被 Task43 命中）

结论：这类任务的主要难点不在“正则是否够严”，而在“跨类型冲突仲裁 + 证据闭环”，否则会出现系统性漏检。

## 6. 准确度能否达到 99%？（评估与条件）

### 6.1 在什么定义下可以达到/接近 99%
当“准确”定义为：
- 号码存在性（文本中出现）；
- 格式/校验可验证（长度/生日段/校验位/Luhn）；
- 模型输出必须引用证据ID，且经一致性校验；

则对“号码类标签”做到 99%+ 是现实的（尤其你接受慢一点）。

### 6.2 在什么定义下无法保证 99%
当“准确”定义为：
- 号码是否真实存在、是否本人、是否有效证件；
- 文本缺失或被遮挡仍要给出确定结论；
- 所有歧义都必须自动决策且永远正确；

则无法保证 99%（缺外部真值与信息缺失）。

### 6.3 让“99%”真正可落地的验收建议
- 以“可验证标签”优先：先把身份证/银行卡/手机号的存在与格式校验做成稳定链路。
- 对歧义样本允许 `needs_review`：用复核兜底提升总体正确率。
- 按数据集跑指标：每次改动跑回归集与抽样集，追踪 precision/recall/F1 与 needs_review。

## 7. 里程碑计划（建议）

### M0：口径与验收（1-2天）
- 明确每个号码类标签的定义与冲突优先级（含 15位身份证口径）。
- 建立回归集（xlsx/抽样集）与指标计算方式。

### M1：规则证据SSOT（已完成基础能力）
- 统一候选抽取 + 分类校验 + 冲突仲裁 → 输出证据 JSON。
- 否定条件任务（Task43）走证据裁决，消除系统性漏检。

### M2：扩展常用标签（3-7天）
- “身份证是否存在（含错误）”
- “身份证是否错误（按 length / checksum 分类输出）”
- “银行卡号是否存在/是否疑似有效（Luhn）”
- 对每个标签给出：证据字段 → 结论字段 → 一致性校验规则。

### M3：输出结构标准化 + 一致性校验（3-5天）
- 固化 JSON schema（分析/结论/证据引用/needs_review）。
- 落地 `RuleConsistencyValidator`，不一致自动打回/降级。

### M4：端到端回归与上线策略（3-5天）
- 跑全链路回归：导入→任务→结果→导出。
- 增加监控：错误率、needs_review 比例、模型耗时、重试/限流统计。

## 8. 风险与对策（务实版）

- **PII 泄漏风险**：日志/提示词/导出默认脱敏；如需明文仅在权限受控场景下提供。
- **LLM 幻觉风险**：证据闭环 + 禁止输出证据外号码 + 一致性校验（必要时重问）。
- **“真假存在”诉求**：明确为范围外；如业务确需，走内部权威库/接口做二次校验，不交给 LLM。
- **性能风险（可接受慢）**：优先稳定性；批量任务可并发限流+缓存，必要时仅对 needs_review 二次调用。
