# 变更提案: 号码类标签提取与分析链路 99% 准确改造

## 需求背景
当前项目在“警情/接警自由文本”的号码类任务上，存在明显的系统性漏检与可解释性问题，典型表现为：
- 否定条件任务（例如“提取不满足18位的错误身份证号”）大量输出“无”，与文本实际内容不一致。
- 规则预处理结果在提示词中被描述为“100%准确”，但实际存在冲突吞噬、漏检等问题，会误导后续 LLM 推理。

基于 `测试数据/analysis-task-43-results-with-reasoning.xlsx` 的统计（360 行）：
- 现有结果几乎全部为“无”（359/360）。
- 仅用身份证结构特征（地区码首位 1-6 + 生日位 YYYYMMDD 可解析）扫描，至少有 38 行存在“非18位且具有身份证结构”的候选，但当前实现仅命中 1 行，存在系统性漏检。

因此需要将号码类任务的执行链路升级为“规则建模证据（SSOT）→ LLM 分析 → 一致性校验（可选）”，以获得可度量的高准确率与可审计性。

## 产品分析

### 目标用户与场景
- **用户群体:** 警情/接警数据分析人员、数据标注人员、质检人员
- **使用场景:**
  - 批量导入警情文本（Excel/CSV），快速识别“身份证是否存在/是否错误”“银行卡是否存在”等关键字段
  - 对 needs_review（疑难/冲突）样本进行人工复核与修正

### 价值主张与成功指标
- **价值主张:** 用“规则证据模型”把号码判断变成可验证的确定性计算，让 LLM 只做解释与业务分析，避免模型幻觉造成不稳定结论。
- **成功指标（可量化）:**
  - 号码类标签（身份证/银行卡/手机号/错误身份证）在回归集上 **precision ≥ 99%**（以“规则可验证准确”为定义）
  - needs_review 占比可控（目标 ≤ 3% 或按业务要求配置）
  - 对每条结论可输出证据链（候选、校验、冲突处理、最终裁决）

### 人文关怀
- 严格限制 PII 暴露：日志、提示词、导出、前端展示默认脱敏；对“原文片段”仅保留必要上下文并可配置关闭。

## 变更内容
1. 建立“号码证据建模”统一中间层：输出结构化 JSON（脱敏、可审计），作为号码类任务的 SSOT。
2. 修复/替换现有冲突吞噬逻辑：避免“银行卡范围先占用导致身份证候选被跳过”的系统性漏检。
3. 统一 LLM 提示词约束：LLM 只能基于证据 JSON 引用候选，不得凭空生成号码；输出必须可校验。
4.（可选）二次强化：以一致性校验为主（规则校验/证据引用校验），LLM 审核仅用于少量歧义样本。

## 影响范围
- **模块:** backend（必需），frontend（可选展示与复核增强）
- **文件:** 主要涉及规则提取/预处理、提示词构建、结构化提取器、任务执行与结果存储
- **API:** 可选新增字段（例如返回证据 JSON/needs_review 原因）
- **数据:** 可选新增/标准化 `extracted_data`（证据 JSON），或新增专用字段/表

## 核心场景

### 需求: 否定条件号码任务稳定提取（以 Task43 为代表）
**模块:** backend

#### 场景: 提取“不满足18位的错误身份证号”
- 条件：原文中出现疑似身份证结构的数字串（例如 17/19/22 位）或 15 位旧版号（是否纳入由业务定义）
- 预期结果：
  - 规则预处理输出候选列表与证据（命中位置、归因、冲突处理）
  - 结果不再被银行卡/其他数字串规则吞掉

### 需求: 身份证存在/错误/银行卡存在等常用标签可复用
**模块:** backend

#### 场景: 输出派生结论字段
- 预期结果：从证据 JSON 派生 `id_exists`、`id_invalid_non18`、`id_invalid_checksum`、`bank_card_exists` 等，支持直接作为分类标签的判定依据或提供给 LLM。

## 风险评估
- **风险:** 规则提取与 LLM 结果不一致时如何裁决
  - **缓解:** 以规则证据为准；LLM 输出必须引用证据ID；不一致直接 needs_review
- **风险:** PII 泄露（日志/提示词/导出）
  - **缓解:** 默认脱敏；关键路径强制脱敏；提供开关与审计
- **风险:** “99%”指标缺乏真值集导致不可验证
  - **缓解:** 建立回归标注集（抽样+人工复核），持续度量与迭代
